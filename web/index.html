arm64golf — can open weights match AlphaDev on ARM?

In 2023, AlphaDev established a learned-search frontier for x86 sorting routines, reporting Sort3AlphaDev at 17 instructions [Fawzi et al., Nature 618, 257–263]. No equivalent public result exists for ARM64. arm64golf asks whether open-weight coding models, served on consumer Apple Silicon via the MacProvider network, can produce verifiable ARM64 routines shorter than published references. The metric is instruction count over a deterministic 1200-case verifier. The unit of progress is a signed candidate.

Best verified ARM64 sort3: 12 instructions — prompt-assisted, not a discovery-class result. Live MacProvider catalog as of 2026-06-07: 1 provider, 1 model. The benchmark grows as the network does; join as a provider or prompt contributor.

How it works

The harness submits a prompt to an open-weight coding model through an OpenAI-compatible endpoint pinned to a single MacProvider node. The v0.1 configuration is mlx-community/Qwen2.5-Coder-7B-Instruct-4bit on provider air5. Each model response is parsed for an ARM64 assembly block, assembled into a native binary, and executed under a deny-by-default sandbox-exec profile against 1200 deterministic test cases. Any failure rejects the candidate. A passing candidate is hashed, scored by static instruction count after a normalization pass, and bound into an ed25519 receipt that commits to (problem_id, candidate_hash, score, model_id, provider_id, harness_version, timestamp). The public score for a problem is the lowest verified instruction count on record.

The receipt-signing key is operator-held and never sees model output. Anyone with receipts/PUBKEY and the harness source can re-derive any score on the leaderboard from the recorded receipt and the candidate bytes. The leaderboard is therefore a public log of replayable claims, not a ranking of submissions.

Will you beat AlphaDev?

AlphaDev's reported Sort3AlphaDev at 17 instructions targets x86. The instruction set, register file, and conditional-execution primitives of ARM64 differ enough that the comparable ARM64 frontier must be measured separately. The published baseline currently in the harness, 726c3e4c49b5, scores 18 instructions on ARM64.

The open question is whether a coding model in the small open-weight class (≤8B parameters, 4-bit quantized, served on consumer Apple Silicon) can produce a verifiable ARM64 routine that improves on the published ARM64 baseline. A separate, stronger question is whether such a routine improves on the AlphaDev x86 result when normalized for ISA. The benchmark is open to any participant willing to point a provider node and a model at the harness.

Participate

Two roles are open in the v0.1 preview. A third opens after public launch.

Run a provider
Operate an Apple Silicon Mac as an MLX inference endpoint on the MacProvider network. The arm64golf harness, and other workloads, can pin to your node by provider identifier. Provider onboarding.
Register a contestant model
Public contestant intake is closed in v0.1 to preserve attribution between prompt and model during the calibration phase. Submit an interest issue specifying model, quantization, and target provider for post-launch intake. Contestant interest issue.
Propose a problem
The pipeline is problem-agnostic. A candidate problem requires a deterministic verifier, a published reference score, and a justification for instruction-count as the relevant metric. Open a problem proposal.

Methodological note. The best verified ARM64 candidate at 12 instructions (57b2aa236342) is produced by the model tiling the four-instruction cmp/csel/csel/mov block supplied as a worked example in the csel_hint prompt template. The receipt is valid against the 1200-case verifier. The structural credit is shared between the prompt and the sampler. A discovery-class result requires either a verified candidate below 12 from a template that does not contain the structure, or a verified candidate with non-isomorphic structure. Neither has been observed across 223 post-canary evaluations on the 7B configuration. Full record in REPORT.md.
