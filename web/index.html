arm64golf — can open weights match AlphaDev on ARM?

In 2023, AlphaDev established a learned-search frontier for x86 sorting routines, reporting Sort3AlphaDev at 17 instructions [Fawzi et al., Nature 618, 257–263]. No equivalent public result exists for ARM64. arm64golf asks whether open-weight coding models, served on consumer Apple Silicon via the MacProvider network, can produce verifiable ARM64 routines shorter than published references. The metric is instruction count over a deterministic 1200-case verifier. The unit of progress is a signed candidate.

Best verified ARM64 sort3: 12 instructions (Qwen-Coder-7B on csel_hint) and 13 instructions (Llama-3.2-3B on the neutral no_failed_context, no embedded hint). Live MacProvider catalog as of 2026-06-07: 2 providers, 2 models. Add yours.

How it works

The harness submits a prompt to an open-weight coding model through an OpenAI-compatible endpoint on MacProvider. With --allow-marketplace-attribution, the harness can pin to any (provider, model) pair on the network; the v0.1 default pin to air5 + mlx-community/Qwen2.5-Coder-7B-Instruct-4bit remains the conservative default for unattended runs. Each model response is parsed for an ARM64 assembly block, assembled into a native binary, and executed under a deny-by-default sandbox-exec profile against 1200 deterministic test cases. Any failure rejects the candidate. A passing candidate is hashed, scored by static instruction count after a normalization pass, and bound into an ed25519 receipt that names the (provider, model, template) it came from: kind='reference-harness', with signed template_id details. The public score for a problem is the lowest verified instruction count on record.

The receipt-signing key is operator-held and never sees model output. Anyone with receipts/PUBKEY and the harness source can re-derive any score on the leaderboard from the recorded receipt and the candidate bytes. The leaderboard is therefore a public log of replayable claims, not a ranking of submissions.

Will you beat AlphaDev?

AlphaDev's reported Sort3AlphaDev at 17 instructions targets x86. The instruction set, register file, and conditional-execution primitives of ARM64 differ enough that the comparable ARM64 frontier must be measured separately. The published baseline currently in the harness, 726c3e4c49b5, scores 18 instructions on ARM64.

The open question is whether a coding model in the small open-weight class (≤8B parameters, 4-bit quantized, served on consumer Apple Silicon) can produce a verifiable ARM64 routine that improves on the published ARM64 baseline. A separate, stronger question is whether such a routine improves on the AlphaDev x86 result when normalized for ISA. The benchmark is open to any participant willing to point a provider node and a model at the harness.

Participate

Phase 4 moved the project from a single-node calibration preview to an attributable marketplace record. There are now two active recruiting tracks and one gated submission track.

Track A: Run a provider
Operate an Apple Silicon Mac as an MLX inference endpoint on the MacProvider network. The arm64golf harness, and other workloads, can pin to your node by provider identifier. When the harness reaches your node, its responses are bound to your provider_id in every receipt and every pairs[] row. Llama-3.2-3B on air8gb is the current example. Provider onboarding.
Track B: Contribute a prompt template newly open
Phase 2 receipt v2 signs template_id and template_name into every reference-harness receipt, so a contributor's prompt template is attributable in the public record. See prompt template contributing, then submit your template. Track B is operator-reviewed: contributions land via issue, the operator registers the template into harness/prompts.py, runs the canary, and publishes the resulting pairs[] row + receipts. When a template lands, the operator runs a bounded canary targeting roughly 200 candidate responses when provider quota permits; the exact evaluated count is published in pairs[].evaluated_responses. Reference-harness receipts are signed on any verified candidates the template produces.
  1. Open an issue with the template body, intent, and suggested name.
  2. The operator registers it in harness/prompts.py.
  3. The operator runs a bounded canary.
  4. The operator publishes results and signs reference-harness receipts.
The contributor identity in the issue link is credit metadata, NOT cryptographic provenance. The receipt's kind='reference-harness' attestation does not name the contributor; it names the operator-run harness.
