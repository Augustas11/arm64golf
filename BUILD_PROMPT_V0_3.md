## Why v0.3 exists

The canary was never a clean test. Every time it hit a defect, the
operator (us) reached for a production-config dial. That's a useful
diagnostic activity but it confounds two questions:

1. Does the search loop find correct ARM64 with `failed_context`?
2. Would a real buyer running this workload succeed without operator
   intervention?

v0.3 closes the second question by:

1. Capping per-call output deterministically in the harness so the
   60s timeout dials become a fallback rather than a load-bearing
   knob
2. Running a clean canary on the current production config and pinning
   the result as the v0.3 baseline
3. Generating a final REPORT.md that distinguishes "the search works"
   (PASS-A real) from "the network supports the workload" (no operator
   intervention required mid-run)

### Known unresolved variable: air5 reliability

Across v0.1 + v0.2 sessions, air5 (M4 MacBook Air, 16GB, phase3-binary
v1.2.5) was observed entering a "ghost connected" state ~30 minutes
after each restart:

- coordinator pool snapshot shows the provider as `ready` with
  `slots_free=1`, heartbeat continuing
- but live inference requests return 503 because the WS gets torn down
  mid-relay and the provider's response arrives at a closed pipe
- restart of phase3-binary on air5 clears the state for ~30 minutes,
  then the pattern returns

This recurred AFTER the 60s timeout fix was deployed, so it is a
separate failure mode from the 10s timeout cliff. Likely root cause is
in phase3-binary v1.2.5 (the prior v1.2.4 build was tagged
"verbose-keepalive" suggesting that area was specifically worked on).
Operator triage is outside arm64golf's scope.

For v0.3, the air5 issue means a 200-call canary may not complete in a
single sitting even with max_tokens=256. Two acceptable outcomes:

- Canary completes within one ~30 min window (air5 stays healthy) →
  REPORT.md is the v0.3 baseline
- Canary trips the three-consecutive-503 halt → REPORT.md documents
  partial state + the air5 ceiling as the binding constraint, NOT a
  v0.3 failure. The honest signal is "the search loop is fine; the
  provider runtime is not yet production-ready for sustained batch
  load."

In neither case should the operator restart air5 mid-run to "salvage"
the canary. That would re-introduce the operator-in-the-loop
confounder v0.3 exists to remove.
